## US-T01: 無料トライアル週でサービスを試す

**Story (Connextra)**:
> **As a** タロウ（食生活の意思決定を外注したい在宅勤務エンジニア）,
> **I want to** タコ中の機能を 1 週間無料で試してから本契約に入りたい,
> **so that** 強制発注に巻き込まれる前に、サービスの雰囲気と自分の摂取記録を把握できる。

**Priority**: P1
**FR Mapping**: FR-5（ユーザー認証・トライアル）
**INVEST Check**: Independent ✅ / Negotiable ✅ / Valuable ✅ / Estimable ✅ / Small ✅ / Testable ✅

**Acceptance Criteria**:
- **Given** ユーザーがまだサインアップしていない状態で,
  **When** ChatGPT カスタム GPT 共有リンクまたはダッシュボード URL からサインアップする,
  **Then** Cognito Hosted UI で認証され、初回ログイン時に「タコ中契約書」モーダルが表示される。
- **Given** タコ中契約書のモーダルが表示されている,
  **When** ユーザーが「同意してトライアル開始」を押す,
  **Then** トライアル開始時刻が DB に記録され、ダッシュボードで残り日数（例: "Free trial: 7 days left"）が表示される。
- **Given** トライアル期間中（残り日数 > 0）,
  **When** 金曜夜 21:00 の強制発注スケジューラが起動する,
  **Then** 当該ユーザーは強制発注対象から除外され、Push 通知も飛ばない。
- **Given** トライアル期間が満了した次の月曜 00:00,
  **When** ユーザーがダッシュボードを開く,
  **Then** "本契約開始: 来週月曜から強制発注ループが始まります" の通告モーダルが表示される。

## US-T02: タコス摂取を記録する

**Story (Connextra)**:
> **As a** タロウ,
> **I want to** タコスを食べたタイミングを 1〜2 タップで記録したい,
> **so that** 強制発注ロジック（逆比例）と未摂取検知（サルサ通知）が正しく動き、無駄な通知に責められない。

**Priority**: P1
**FR Mapping**: FR-1.3
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** ユーザーがダッシュボードを開いている,
  **When** ヘッダー右上の「TACO!」ボタンを 1 タップする,
  **Then** 現在時刻で摂取イベントが DB に記録され、ダッシュボード上の "今週の摂取回数" が +1 され、最終摂取日時が更新される。
- **Given** ユーザーが摂取量を細かく入れたい,
  **When** 「TACO!」ボタンを長押し（または "詳細記録" を選ぶ）,
  **Then** 日時／個数（1〜10）／自由記述メモを入れるフォームが開き、保存できる。
- **Given** 同一日時に重複した摂取記録を試みる,
  **When** 1 分以内に 2 回連続でタップする,
  **Then** 2 回目はスナックバーで "タコス記録済み（1 分以内の重複は無視）" と表示され、DB は更新されない。

## US-T03: 金曜夜の強制発注通告を受け取る

**Story (Job Story)**:
> **When** 金曜の夜 21:00、タロウが家でくつろいでいるとき,
> **I want to** 来週の食卓が外圧で勝手に決まったことを劇的に通告されたい,
> **so I can** "決めなくていい" 解放感（と諦め）を体験できる。

**Priority**: P1
**FR Mapping**: FR-2.1（逆比例ロジック）, FR-2.2（メニューランダム化）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** 本契約中のユーザーで、過去 7 日の摂取回数 N がある,
  **When** 金曜夜 21:00 の EventBridge スケジューラが起動する,
  **Then** Lambda が `BASE + max(0, THRESHOLD - N) * MULTIPLIER` で来週の発注量を算出し、メニュープールからランダム選定する。
- **Given** 発注量と発注内容が決定された,
  **When** その時点でユーザーがオンライン（PWA 起動中），
  **Then** ダッシュボード上にフルスクリーンモーダルが表示される（タイトル "今週の食卓、決定" / 発注量 / メニュー / 配送日 / "キャンセルできません" の文言）。
- **Given** 発注量と発注内容が決定された,
  **When** ユーザーがオフライン,
  **Then** Web Push 通知が即座に送られ、本文に "アミーゴ！来週は X 人前のタコスが届くぞ" と表示される。
- **Given** 通告モーダル／通知の Push,
  **When** いずれかのチャネルでユーザーがアクションを起こす（モーダル「了解」 or 通知タップ）,
  **Then** ダッシュボード "来週の食卓" セクションが更新済み状態として表示され、配送ステータスが `decided` になる。

## US-T04: 「キャンセル不可」を試そうとして失敗する

**Story (Job Story)**:
> **When** 強制発注通告を見たタロウが「これ本当にキャンセルできないの？」と思ったとき,
> **I want to** キャンセル試行が**逆に発注量を増やす**シュールな反応を体験したい,
> **so I can** タコ中の世界観を肌で理解し、笑いながら諦める。

**Priority**: P1
**FR Mapping**: FR-2.3（キャンセル不可表現）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** "来週の食卓" セクションがダッシュボードに表示されている,
  **When** ユーザーがそのセクションを表示する,
  **Then** "キャンセル要求" ボタンが表示されている（押せる状態）。
- **Given** ユーザーが「キャンセル要求」ボタンを押す,
  **When** 押下確認モーダルが表示され、ユーザーが "OK" を押す,
  **Then** 来週の発注量が +2 人前され、UI 上で「キャンセル要求を承りました。お詫びとしてサービスで 2 人前を追加いたしました。来週月曜、無事配送いたします。」というシュールなトーストが表示される。
- **Given** 同じ週内で複数回キャンセル試行する,
  **When** 2 回目以降のキャンセルボタンを押す,
  **Then** 押下するたびに +2 人前され、上限 20 人前で頭打ち。20 人前到達時のトースト：「もうこれ以上は冷蔵庫が破裂します」。

## US-T13: 月曜夜に材料受け取り + Bedrock レシピ配信（新規・v1.1）

**Story (Job Story)**:
> **When** 月曜の夜 21:00、来週分の材料が玄関に届いたとき,
> **I want to** ダッシュボードに材料セットの内容が表示され、Bedrock 生成のアミーゴ口調レシピがそのまま PWA と Web Push で配信されてほしい,
> **so I can** 「料理から逃げてきた」自分でも、レシピがあるから作るしかない状態に追い込まれる。

**Priority**: P1
**FR Mapping**: FR-2.3（レシピ配信）, FR-2.4（配送モック）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** 本契約中で `decided` 状態の発注がある,
  **When** 月曜 21:00 の EventBridge スケジューラが起動する,
  **Then** 配送ステータスが `delivered` に遷移し、ダッシュボードに "材料が届きました（モック）" カードが表示される。カードにはキット種別（Type A: タコス専用キット / Type B: 食材ばら売り）と人前数が大胆タイポで表示される。
- **Given** 配送ステータスが `delivered` になった,
  **When** Lambda が Bedrock を呼び、当該キット構成（人前・主肉・サルサ・テーマ）を渡す,
  **Then** Bedrock が "アミーゴ口調" のレシピを生成し DB に保存する。
- **Given** Bedrock 呼び出しが失敗する（タイムアウト・エラー）,
  **When** Lambda が DLQ にイベントを送り、フォールバック静的レシピ 5 本からランダム選定する,
  **Then** ダッシュボードはフォールバックレシピを表示し、ユーザー体験は崩れない。
- **Given** レシピ生成が完了している,
  **When** Web Push 通知を送る,
  **Then** "アミーゴ！材料が玄関に届いたぞ。今週は X 人前 PASTOR。レシピを開け" のメッセージが Push され、タップで PWA の "今週の調理ガイド" カードへ遷移する。
- **Given** ユーザーが同日中にレシピを再表示する,
  **When** "今週の調理ガイド" カードを開く,
  **Then** キャッシュされたレシピが即座に表示される（再生成しない、コスト最適化）。

## US-T14: 24 時間カウントダウンタイマーで台所に追い込まれる（新規・v1.1）

**Story (Job Story)**:
> **When** 月曜夜に材料を受け取ったタロウが「明日でいいや」と思ったとき,
> **I want to** ダッシュボードに残り時間を秒単位で示すカウントダウンタイマーがあり、Tシャツ罰の発火時刻（火曜 21:00）が常に視界に入ってほしい,
> **so I can** 「逃げ場のない時間圧」を体感し、結局台所に立つ。

**Priority**: P1
**FR Mapping**: FR-1.2（24h ルール連動）, FR-4.1（ダッシュボードカウントダウン）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** 配送ステータスが `delivered` で「作った」記録がない,
  **When** ユーザーがダッシュボードを開く,
  **Then** トップに **24 時間カウントダウンタイマー**（HH:MM:SS、大胆タイポ）が表示され、火曜 21:00 まで秒単位で減算される。
- **Given** カウントダウン残り 6 時間（火曜 15:00）,
  **When** タイマー UI が更新される,
  **Then** タイマー色がオレンジに変わり、警告アイコン（SVG）が並ぶ。
- **Given** カウントダウン残り 1 時間（火曜 20:00）,
  **When** バックエンドが Web Push を送る,
  **Then** "アミーゴ、残り 1 時間だ。Tシャツの発注書は既に印刷待ちだ" のメッセージが Push される。
- **Given** ユーザーが期限内に「TACO!（=作った）」を記録する,
  **When** バックエンドがカウントダウンを停止する,
  **Then** タイマー UI が消え、"アミーゴ、よくやった" の達成バナーが 5 秒表示される。

## US-T07: 24h ウィンドウ超過時、Tシャツ罰と同時にサルサ通知が鳴り続ける（v1.2 改修）

**Story (Job Story)**:
> **When** タロウが月曜夜に届いた材料を 24 時間以内に作らず、火曜 21:00 のチェックポイントに突入したとき,
> **I want to** Tシャツ発注の通告と同じ瞬間に、サルサ音の Web Push が鳴り続けて無視できない状態にされたい,
> **so I can** 「火曜に食べる」という強制サイクルを身体で覚え、即座に摂取記録を入れて沈静化する達成感を得る。

**Priority**: P1
**FR Mapping**: FR-1.1（v1.6 で再定義）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** 配送ステータスが `delivered` のまま、かつ delivered 後 24 時間以内に「作った」記録がない,
  **When** 火曜 21:00 のチェックポイント Lambda が起動する,
  **Then** Tシャツ発注（FR-1.2 / US-T06）と**同時に**サルサ通知ループの開始フラグが ON になる。
- **Given** サルサ通知ループ ON,
  **When** バックエンドが 30 分間隔で Web Push を送る,
  **Then** Push は `notification.requireInteraction=true` で送られ、ユーザーが操作するまで通知センターに残る（鳴り続ける体験）。
- **Given** サルサ通知が鳴っている,
  **When** ユーザーが通知をタップしてダッシュボードで「TACO!」ボタンで摂取を記録する,
  **Then** バックエンドはサルサループを OFF にし、ダッシュボードのサルサ警告バナーが消え、FR-6.4 のエスカレーションカウンタも 0 リセットされる。
- **Given** サルサ通知が鳴っているがユーザーが Push をタップせず通知センターから削除のみ,
  **When** 30 分後の次サイクルが起動する,
  **Then** サルサ通知が再送される（無視できないループ）。
- **Given** v1.5 までの「3 日連続未摂取」ルール,
  **When** v1.6 に移行する,
  **Then** 旧ルールは廃止され、サルサ通知の発火条件は **「24h カウントダウン超過時のみ」** に統一される。

## US-T08: 平日昼夕にメキシコ風誘惑 Push を受け取る

**Story (Connextra)**:
> **As a** タロウ,
> **I want** 平日昼 12:00 と夕 18:00 に、メキシコ風キャラから誘惑 Push が届くこと,
> **so that** 食事決定の負担を取り除き、タコス摂取の選択肢を強く意識できる。

**Priority**: P1
**FR Mapping**: FR-6.2（誘惑 Push 中程度レイヤー）
**INVEST Check**: I ✅ / N ✅ / V ✅ / E ✅ / S ✅ / T ✅

**Acceptance Criteria**:
- **Given** 本契約中で Web Push 購読済みのユーザー,
  **When** 平日 12:00 ±10 分のスケジュールジョブが起動する,
  **Then** Bedrock の Claude が「今日のタコス煽り文（メキシコ風キャラトーン）」を生成し、Web Push 通知を送る（例: "アミーゴ！昼だ。冷蔵庫のトルティーヤが待ってるぞ"）。
- **Given** 同上,
  **When** 平日 18:00 ±10 分のジョブが起動する,
  **Then** 別バリエーションの誘惑文が生成され、夕方版として送られる。
- **Given** 強制バリエーションウィーク発動中,
  **When** 上記の昼／夕 Push が走る,
  **Then** Push 文に "今週はマンゴーウィークだ" 等のテーマがインジェクトされ、トーンが激化する。

## US-T09: 24h カウントダウンウィンドウ内で Push 無視回数に応じてトーンがエスカレートする（v1.2 改修）

**Story (Job Story)**:
> **When** タロウが月曜 21:00 の材料受領後、火曜 21:00 までの 24 時間ウィンドウ内に届く誘惑 Push を順次無視していくとき,
