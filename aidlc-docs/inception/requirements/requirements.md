## 1. Intent Analysis Summary

### 1.1 User Request
タコス依存症の重症化を「優しく加速」する PoC サービス「タコ中」を開発する。

**ビジネス意図の核心**:
英語圏に根付いた「Taco Tuesday（タコスチューズデー）」——火曜日にタコスを食べる週次文化——は、日本にはまだ存在しない。本サービスの**文化的使命**は、AWS サーバーレスと AI の力を借りて、**Taco Tuesday を日本の生活リズムに移植すること**にある。

強制発注・AI 煽り文・Tシャツ罰・サルサ通知など複数の依存ループ要素を組み合わせ、外圧でユーザーを「作らせる」仕組みを中心に設計する。「人をダメにする」とは、タコスが**頭の中に住み着く（認知の占有）**状態——3 ヶ月後に「タコスのことばかり考えている。最高じゃないですか」と感じる幸せな降伏を指す。

**AWS Summit Japan 2026 AI-DLC ハッカソン**（テーマ「人をダメにするサービス」）への応募作品。

### 1.2 Request Classification
| 項目 | 評価 |
|------|------|
| **Request Type** | New Project (Greenfield) |
| **Scope Estimate** | Cross-system（AI 刺激生成、通知、配送モック、AWS サーバーレス） |
| **Complexity Estimate** | Complex（複数統合点） |
| **Requirements Depth** | Standard |

### 1.3 Concept Statement

> タコスは偉大である。本サービスは、ユーザーのタコス欲を**予測しない**。**生成し、煽り、強制する**。
> **コア・コンセプト**: 「**タコスが生活の一部になる**」。
> 完成品ではなく**材料**が届き、ユーザー自身が手を動かしてタコスを作る。
> 作らなければ罰（Tシャツ送付・サルサ通知）が降り、平日の昼夜は誘惑 Push がスマホを揺らす。
> 食事だけでなく**台所・冷蔵庫・通知音・SNS の話題**まで、タコスが日常に侵食する。
> 世界初の積極的依存支援プラットフォームの **PoC**。
> （免責: 医療機器ではない／パロディ的要素を含む）

**v2.0 深掘り — なぜタコスか / 「ダメになる」とは何か:**

> **なぜタコスなのか。**
> 「Taco Tuesday（タコスチューズデー）」——毎週火曜日にタコスを食べる——という文化が英語圏に根付いているにもかかわらず、日本にはまだ存在しない。タコ中の使命は、AWS サーバーレスと AI の力を借りて、**Taco Tuesday を日本の生活文化として移植すること**にある。プロジェクト名「tako-tues」はこの宣言そのものだ。
>
> **「人をダメにする」とは何か。それは認知の占有である。**
> タコスが「頭の中に住み着く」状態——Push を受け取れば翌朝もタコスを考え、材料が冷蔵庫に入れば週末もレシピを調べてしまう。3 ヶ月後、ユーザーは気づく。「あれ、私、タコスのことばかり考えている。**最高じゃないですか**」
>
> これは**幸せなダメになり方**である。依存は苦しくない。むしろ心地よい。「ダメになる」とは、自分の生活リズムがタコスに支配され、それが**嫌いではない**状態のことだ。

### 1.4 設計原則 (Design Principles)
本プロジェクトのすべての機能は、以下の原則に整合していること。

1. **文化移植 ＞ 単発体験**: 1 回のタコス摂取で終わらず、「火曜 = タコス」という週次リズムが日本文化に根付くことを最終目標とする。すべての機能はその積み重ねの一コマである
2. **頭に住み着く ＞ 腹を満たす**: 食欲を満たすだけでなく、タコスが「考えない日がない」存在になることを目指す（認知占有）。Push・煽り文・レシピ配信がすべてこの認知占有を強化する
3. **強制 ＞ 提案**: ユーザーの欲を「当てに行く」のではなく「外圧で作り出す」。Push・発注トリガー・Tシャツ罰がユーザーを動かす——ユーザーは能動的に選んでいない（コントロール喪失の笑い）
4. **作らせる ＞ 食べさせる**: 完成品ではなく材料を届け、調理という時間と労力をユーザーに投じさせる。手を動かす行為が習慣化の鍵であり、「作った」という体験がリズムを刻む
5. **幸せな降伏**: 「ダメになる」は苦痛ではなく快感。「最高じゃないですか」というトーンで依存を肯定・祝福する。このサービスはユーザーの敵ではなく、幸せな共犯者
6. **逃げ場をなくす**: 食べない／作らない選択肢を取りづらい状況をデザインする（罰・エスカレーション・「届いた材料を捨てる罪悪感」）
7. **AI は欲を生成する側**: AI/Claude は予測器ではなく **刺激生成器・調理サポーター** として使う。欲求を「発見」するのではなく「製造」する

## 2. Project Scope and Direction

### 2.1 プロジェクト規模
- **スコープ**: PoC（概念実証） — 「依存ループ」のメイン体験を実演できるプロトタイプ。データはダミーで OK。
- **ターゲットユーザー**: 一般消費者（タコス愛好家）想定。ただし**まずは作者本人向け**に作り、将来的に一般公開を視野に入れる。
- **デプロイ先**: AWS フル活用（**ハッカソン規約により AWS 上での稼働が必須**）。サービス選定は機能要件・開発速度・コンセプト適合を優先し、無料枠縛りはしない。
- **リリース目標（書類審査）**: **2026-05-10 23:59 JST まで** — Inception 成果物 + 公開 GitHub
- **リリース目標（予選 MVP）**: **2026-05-30** — 動作する MVP デモ
- **リリース目標（決勝）**: **2026-06-26** — AWS 上にデプロイされた動作デモ
- **成功基準（一次）**: 書類審査通過（15チーム選出）→ 予選通過（5チーム選出）→ 決勝入賞
- **成功基準（二次）**: 友人・SNS でデモして話題になる（バイラル／話題性）

### 2.2 想定外（Out of Scope）
- 実際の物販・配送（Shopify / Uber Eats / Printful の本番発注）
- 大規模なユーザーベース（>100 ユーザー）
- 医療データ取扱コンプライアンス（HealthKit 連携は**モック化**）
- iOS ネイティブアプリ（HealthKit 統合は将来検討）
- 完全な決済機能・課金

## 3. Functional Requirements (機能要件)

### FR-1: 強制トリガー機能（**最優先 / コア機能**）

> Q4 で B「作らされる強制トリガー（Tシャツ送付・サルサ通知）」が選択されたため、本機能を MVP のコアとする。
> v1.6 でサルサ通知と Tシャツ罰のトリガーを統合し、**「火曜日に食べる」前提のサイクル**に揃えた。

**FR-1.1 サルサ通知（24h カウントダウン超過時）**:
- 配送ステータスが `delivered`（月曜 21:00）後、**24 時間以内**（火曜 21:00 まで）に「作った」記録がない場合、火曜 21:00 のチェックポイントで Tシャツ罰（FR-1.2）と**同時にサルサ通知ループ**が発火する。
- サルサ通知は Web Push を 30 分間隔でループ送信し、`requireInteraction=true` で**鳴り続ける**。
- ユーザーが摂取記録を入力（FR-1.3）した時点でループが停止する。
- v1.5 までの「3 日ルール」は廃止 — 「火曜に食べる」前提で月〜火の 24h ウィンドウに統合。

**FR-1.2 Tシャツ強制送付（モック）**:
- 配送ステータスが `delivered` 後**24 時間以内に「作った」記録がない**場合、火曜 21:00 のチェックポイントで発火する。
- Tシャツ（黒地に TACO×3）を**発注済み**通知をユーザーに表示する。
- 実際の Print-on-Demand 発注は行わず、UI 上で「✅ 発注済み」状態と擬似配送ステータス（〜5日後に到着予定）を表示する。
- **FR-1.1 と同時発火**（火曜 21:00 単一トリガーで両方発動）。

**FR-1.3 タコス摂取記録（=作った記録）**:
- 1 ステータス制: ユーザーは「**作った（=食べた）**」イベントを 1〜2 タップで記録する。
- 「材料受け取り済み → 作った」のフラグだけ管理し、調理中／完成のような中間ステータスは持たない（PoC スコープ最適化）。
- 摂取履歴は DB に永続化される。
- 火曜 21:00 までに本記録があれば FR-1.1 / FR-1.2 は不発、FR-6.4 のエスカレーションカウンタも自動でリセットされる。

### FR-2: 強制発注エンジン（**完成品ではなく "材料" を届ける**）

> **設計原則**: ユーザーのタコス欲を予測するのではなく、外圧で「来週の食卓」を勝手に決定する。
> 届くのはタコス完成品ではなく**材料**。ユーザーは自分で台所に立たされる。
> ロジックは **「逆比例 + ランダム驚き」のハイブリッド**。

**FR-2.1 発注量決定ロジック（逆比例ベース）**:
- 金曜夜 21:00 のスケジューラ起動時、過去7日間の摂取（=作った）回数 N を取得する。
- 来週の発注量 = `BASE_PORTIONS + max(0, THRESHOLD - N) * MULTIPLIER`
  - 例: BASE=4, THRESHOLD=5, MULTIPLIER=2 のとき、先週 0 回 → 14 人前、先週 5 回以上 → 4 人前
- **食べないほど来週は地獄になる**ことをロジックで保証する。

**FR-2.2 配送内容（材料セット）**:
- 通常週は **タコス専用キット (Type A)**: トルティーヤ + 主肉（パストール／カルニタス／フィッシュ等）+ サルサ + 簡易トッピングが**ほぼ組み立てるだけ**で完成する構成。
- 一定確率（例: 10%）で「**強制バリエーションウィーク**」発動 → 構成が **食材ばら売り (Type B)** に切り替わり、ユーザーが本格的な調理工程をこなす必要がある（例: 「全部マンゴー味」「全部ハバネロ MAX」「丸ごと豚肩肉から作るカルニタス」等）。
- メニュープール: ベジ／フィッシュ／ビリア／アル・パストール／ブレックファスト／マンゴー／ハバネロMAX／カルニタス・スクラッチ等。
- 結果として「来週何が、どの調理難度で届くか分からない」緊張感を作る。

**FR-2.3 レシピ配信（必須・v1.7 で再定義）**:
- 材料セットには**毎回レシピ**が同梱される。
- 配信タイミング: 配送ステータスが `delivered`（月曜夜）になった瞬間、PWA と Web Push の両方でレシピを配信する。
- **レシピソース**: **リポジトリ同梱の静的レシピライブラリ**（`assets/recipes/*.json`）から、キット種別と 1:1 で対応するレシピを決定論的に選定する。
  - 例: Type A・パストール → `assets/recipes/type-a-pastor.json`
  - 例: Type B・全部マンゴー → `assets/recipes/type-b-mango-week.json`
- **レシピは事前に手書きでキュレーションされ、アミーゴ口調で執筆される**（"アミーゴ、まずはトルティーヤを温めるんだ。フライパンで両面 30 秒、それ以上は焦げる。"）。
- **動的生成は行わない**。理由: ユーザーが実際に調理して摂食するため、生成 AI のハルシネーションによる誤った調理手順・分量・火加減指示を完全に排除する必要がある。
- 更新は不定期（リポジトリへの PR で運用）、毎週の自動再生成は無し。
- レシピ表示はダッシュボード "今週の調理ガイド" カードで参照できる。

**FR-2.4 発注フロー（モック実装）**:
- 実 API は呼ばず、DB 上で配送ステータスを以下のように遷移させる:
  - 金曜夜 21:00: `decided` （発注内容決定 + ユーザー通知）
  - 月曜夜 21:00: `delivered` （材料が玄関に届いた状態 + レシピ配信）
  - 火曜夕 21:00: チェックポイント発火（FR-1.2 と連動、24 時間ルール）
- ユーザーは発注内容を**キャンセルできない**（強制性の担保）。一方で「作った」記録は自由に入力可。

**FR-2.5 段階的拡張ロードマップ（参考）**:
- Phase 1（PoC / MVP）: 完全モック
- Phase 2（予選後）: Shopify / Uber Eats サンドボックス API 統合（材料セット手配を実 API に接続）
- Phase 3（決勝後）: 本番 API（対象外）

### FR-6: タコス欲刺激エージェント（**AIが欲を生成する**）

> **設計原則**: AI は予測器ではなく**刺激生成器**として動く。控えめ（FR-6.1）と中程度（FR-6.2）の 2 レイヤーで、AWS サーバーレス内で完結させる。
> v1.8 で過激レイヤー（FR-6.3 ChatGPT カスタム GPT）はスコープから削除。

**FR-6.1 ダッシュボード煽り文句（控えめレイヤー）**:
- ダッシュボードを開くたび、Claude が「今日のタコス煽り文句」を1つ生成して上部に表示する。
- 生成は天気・曜日・直近の摂取記録・配送ステータスを文脈にする。
- 例: 「火曜の雨。雨の日のカルニタスは罪深い」「3日前に食べたあのタコス、もう一回ね？」

**FR-6.2 誘惑 Web Push 通知（中程度レイヤー）**:
- 1 日 2〜3 回、定刻（朝 9:00 / 昼 12:00 / 夕 18:00 など）に Web Push が届く。
- 通知本文は Claude が生成（短文 + 画像URL）、画像は事前生成 or プレースホルダーでよい。
- 通知タップ → ダッシュボード「作った」フォームへ直行。
- **24h カウントダウンウィンドウ内（月 21:00 〜 火 21:00）に発火する Push が、トーンエスカレーションの主要トリガー**となる（FR-6.4 参照）。
- カウントダウン外（水〜金など）の Push は通常トーン（Level A）。

**FR-6.3** ~~ChatGPT カスタム GPT による会話誘導（過激レイヤー）~~ → **v1.8 で削除**
- v1.8 でスコープから削除。AWS ハッカソン作品として AWS 外サービス（OpenAI ChatGPT）依存を排除。
- 過激レイヤーは廃止し、刺激エージェントは FR-6.1（控えめ）+ FR-6.2（中程度）の 2 レイヤー構成とする。
- 関連配布物（`assets/prompts/tako-gpt-*.md`）も配置しない。

**FR-6.4 24h ウィンドウ内 Push エスカレーション仕様（v1.6 で再定義 / v1.8 で範囲修正）**:
- 連続無視カウンタは、配送ステータスが `delivered` になった月曜 21:00 から火曜 21:00 までの **24 時間ウィンドウ内**で集計する。
- カウンタ更新ルール:
  - **+1**: 当該ウィンドウ内に発火した Push 通知をユーザーがタップせず一定時間（例: 60 分）経過した
  - **0 にリセット**: ユーザーが Push をタップ、またはダッシュボードで「TACO!（=作った）」を記録した
  - **次サイクル開始時にリセット**: 火曜 21:00 のチェックポイント終了後、または新しい `delivered` 状態への遷移時
- カウンタ → トーン対応:
  - 0〜1: **Level A（メキシコ風親しみ系・焦り少々）**
  - 2〜3: **Level B（強迫的・罪悪感を煽る）**
  - 4 以上: **Level C（擬人化的に責める）**
- カウンタ N が増えるほど、Push の本文・煽り文句のトーン指示パラメータが昇格する（v1.8 で GPT 説教モード参照を削除）。
- **ユーザー観点では「短時間（最大 24 時間）に集中したエスカレーション」体験**になり、24h カウントダウンタイマーの逼迫と Push トーンの激化が同期する。

**FR-6.5 生成テキストの実装方針**:
- FR-6.1 / FR-6.2 の生成テキストは Bedrock（Claude モデル）または Anthropic API 経由で生成する。
- MVP では一部静的テンプレート + Claude 動的生成のハイブリッドでも可。
- フォールバック静的テンプレートを Level A/B/C ごとに最低 5 本ずつ用意する。

### FR-3: ~~タコス会話縛り~~ → **v1.8 で完全廃止**

> v1.2 で FR-6.3（過激レイヤー）に統合され、v1.4 で実装手段を ChatGPT カスタム GPT に変更したが、
> v1.8 で FR-6.3 自体をスコープから削除したため、本要件も廃止。
> 「会話の入口を侵食する」体験は MVP には含めない。

### FR-4: ダッシュボード（Web）

**FR-4.1 サマリー画面**:
- 今週のタコス摂取（=作った）回数、最終摂取日時、サルサ通知発火可能性、配送ステータスを一覧表示。
- 配送ステータスが `delivered` 中は **24 時間カウントダウンタイマー**を大胆タイポで表示し、Tシャツ罰の発火タイミングをリアルタイム可視化する。

**FR-4.2 今週の調理ガイド**:
- レシピライブラリから決定論的に選定されたレシピを表示する専用カード（FR-2.3 連動）。
- 「アミーゴ口調」のレシピをそのまま閲覧できる。
- **再生成ボタンは無し**（同じキットには同じレシピが対応するため、ユーザーには "今週のレシピ" として一意の存在を提示する）。

**FR-4.3 摂取履歴グラフ**:
- 摂取量と「幸福度（ダミー入力）」の相関を簡易グラフで可視化（HealthKit はモック）。

**FR-4.4 強制トリガー履歴**:
- 過去のサルサ通知発火履歴、Tシャツ「発注」履歴を表示。

### FR-5: ユーザー認証（最小限）

- PoC のため、まずは作者本人のみが利用可能（**シングルユーザー想定**）。
- 簡易認証（Cognito Hosted UI または独自トークン）。コストを優先。

## 4. Non-Functional Requirements (非機能要件)

### NFR-1: パフォーマンス／スケーラビリティ
- **想定ユーザー数**: 〜100 ユーザー（実質シングルユーザー〜小規模）
- **応答時間**: ダッシュボード初期表示 3 秒以内（PoC として許容）
- **スケーリング戦略**: シングルリージョン、サーバーレス前提、水平スケール不要

### NFR-2: コスト方針
- **予算目安**: 月額 数百円〜数千円以内（Q13=A、個人持ち出し）
- **設計方針**: 機能要件・開発速度・コンセプト適合を最優先。無料枠**縛り**は行わないが、必要のない常時稼働や高額サービスは避け、原則サーバーレスを優先する。
  - 採用しやすい候補: Lambda、API Gateway、DynamoDB、S3 + CloudFront、EventBridge Scheduler、Cognito、Bedrock、Step Functions
  - 採用可（必要に応じて）: Aurora Serverless v2、AppRunner、SQS、SNS、SES、CloudWatch Synthetics
  - 必要性を吟味すべき: 常時稼働 EC2、NAT Gateway、ElastiCache（PoC では通常不要）
- **コスト監視**: AWS Budgets で月額 ¥3,000 程度のアラート設定を Construction フェーズで構成する。

### NFR-3: プライバシー／コンプライアンス
- **レベル**: 個人プロジェクト最低限（Q12=C）
- **データ保護**: HTTPS 必須、最低限のシークレット管理（AWS Secrets Manager または Parameter Store）
- **個人医療データ**: HealthKit 連携モックのため対象外
- **HealthKit データ実扱いを将来追加する場合は再アセスメント必須**

### NFR-4: 信頼性
- **SLO**: ベストエフォート（PoC のため 99% 程度を目標）
- **監視**: CloudWatch Logs / Metrics（無料枠範囲内）
- **データバックアップ**: DynamoDB Point-in-time Recovery 等の標準機能のみ

### NFR-5: 開発・保守性
- **言語**: **Python**（Q8=B）
- **テスト**: 重要ロジックには PBT を **Partial** 適用（後述）
- **CI/CD**: GitHub Actions で最低限のテスト・デプロイ

### NFR-6: スピード（最重要）
- **書類審査向け**: 2026-05-10 までに Inception フェーズ成果物 + 公開 GitHub
- **予選 MVP**: 2026-05-30 までに動作する MVP デモ
- **決勝**: 2026-06-26 までに AWS デプロイ済み完成版
- 段階を意識した AI-DLC ステージ実行（書類審査までは Inception 重視、予選までに Construction）

## 6. Extension Configuration

| Extension | Enabled | Mode | Decided At | 備考 |
|---|---|---|---|---|
| Security Baseline | **No** | — | Requirements Analysis | Q14=B、PoC のためスキップ |
| Property-Based Testing | **Yes (Partial)** | Partial（PBT-02, PBT-03, PBT-07, PBT-08, PBT-09 のみ強制） | Requirements Analysis | Q15=不明 → AI-DLC 推奨 (Partial) で仮置き |

> **Q15 についての補足**:
> - **PBT (Property-Based Testing)** は「ランダム入力を大量生成して invariant をチェック」するテスト手法（Hypothesis 等のライブラリ使用）。
> - 本プロジェクトでは「最後にタコス食べてから何日経ったか」「サルサ通知の鳴動条件」など、日付計算やしきい値判定の純粋関数があるため、PBT が役立つ可能性が高い。
> - **Partial** モードなら、純粋関数とシリアライゼーション往復だけが対象になり、PoC のスピード感を損なわず堅牢性を補強できる。
> - もし「やっぱり No（C）でいい」と希望される場合は次のフェーズで切替可能。

## 7. User Scenarios（主要シナリオ）

### Scenario A & B（統合）: 「24h カウントダウン超過 → Tシャツ罰 + サルサ通知 同時発火」シナリオ
1. 月曜夜 21:00、システムが `delivered` に遷移、リポジトリ同梱のキット種別対応レシピが Push 配信
2. ダッシュボードに 24 時間カウントダウンタイマーが起動
3. 月〜火の 24h ウィンドウ内、定刻 Push（朝 9 / 昼 12 / 夕 18）が発火、ユーザーの無視回数で **Level A → B → C にトーン昇格**
4. 火曜夜 21:00 のチェックで「作った」記録がないと判定
5. **同時に発火**:
   - **Tシャツ発注（FR-1.2）** — ダッシュボードに「Tシャツ発注済み」表示、配送タイムラインで「金曜到着予定」
   - **サルサ通知ループ（FR-1.1）** — Web Push が 30 分間隔でループ送信、`requireInteraction=true`
6. ユーザーが「TACO!」を押す（=作って食べた記録）まで、サルサ通知は鳴り続ける
7. 摂取記録時、サルサループ停止 + エスカレーションカウンタ 0 リセット

### Scenario C: 「金曜夜の強制発注」シナリオ
1. 金曜夜 21:00、EventBridge がトリガー
2. Lambda が過去7日間の摂取（=作った）回数 N を取得
3. 逆比例ロジックで来週の発注量を算出（作らないほど大量）+ メニュープールからランダム選定
4. 通常週は **Type A（タコス専用キット）**、10% 確率の強制バリエーションウィークでは **Type B（食材ばら売り）** に切り替わる
5. 配送ステータスを `decided` にして DB 保存、ユーザーに通知（キャンセル不可）
6. 月曜夜、ステータスが `delivered` に遷移し、リポジトリ同梱のキット種別対応レシピが Push 配信される

### ~~Scenario D: 「ChatGPT がタコスの話だけしてくる」シナリオ~~ → **v1.8 で削除**

> FR-6.3 廃止に伴いシナリオも削除。Scenario E 以降の番号は維持する（ID 安定性優先）。

### Scenario E: 「煽り文句と 24h ウィンドウ内 Push エスカレーション」シナリオ（FR-6.1, FR-6.2, FR-6.4）
1. 月曜夜 21:00 の `delivered` 遷移と同時に 24h カウントダウン開始 + 無視カウンタを 0 で初期化
2. ユーザーが朝ダッシュボードを開くと、Claude 生成の「今日のタコス煽り文句」が表示
3. 火曜 9:00 / 12:00 / 18:00 で Push 発火、ユーザーが 60 分以内にタップせず → **カウンタ +1 ずつ昇格**
4. カウンタ 2〜3 で **Level B（強迫的）**、4 以上で **Level C（擬人化）** にトーン変化
5. ユーザー観点では月〜火の限られた時間内に**短時間で激化する Push**を体験する
6. 通知タップで即座にカウンタ 0 リセット + ダッシュボード「作った」フォームへ直行

### Scenario F: 「材料受け取りとレシピ配信」シナリオ（FR-2.3 / v1.7 で再定義）
1. 月曜夜 21:00、配送ステータスが `delivered` に遷移
2. Lambda がキット種別（例: Type A・パストール 4 人前）に対応するレシピを `assets/recipes/type-a-pastor.json` から決定論的に選定
3. PWA に「今週の調理ガイド」カードが出現（手書きアミーゴ口調レシピ）、Web Push で「アミーゴ、材料が届いたぞ。レシピを開け」
4. ダッシュボードに 24 時間カウントダウンタイマーが起動
5. ユーザーは台所に立つしかない

## 9. Key Requirements Summary（要約）

**ビジネス意図（v2.0 深掘り版）**:

> **タコ中が存在する理由**: 英語圏に根付く「Taco Tuesday」文化を、AWS と AI の力で日本に移植する。サービスを通じて毎週火曜日にタコスを作る習慣が日本人の生活に刻み込まれる。
>
> **「人をダメにする」の定義**: タコスが**頭の中に住み着く（認知の占有）**状態。3 ヶ月後に「タコスのことばかり考えている。最高じゃないですか」と感じる幸せな降伏。これは強制（外圧で行動させる）によって達成される。ダメになることは苦痛ではなく快感である。

**機能要件・技術要件（v1.8 継承）**:
1. **AWS Summit Japan 2026 AI-DLC ハッカソン応募作品**（テーマ「人をダメにするサービス」）
2. **書類審査（2026-05-10 締切）→ 予選（5/30）→ 決勝（6/26）** の3段階を見据えた段階的開発
3. **コンセプト = Taco Tuesday の日本化**: 「火曜 = タコスを作る日」を日本の文化として根付かせる
4. **届くのは "材料"**: 完成品ではなく、ユーザー自身が手を動かして作る構造
5. **強制トリガー（サルサ通知＋Tシャツ発注モック）**が MVP のコア (FR-1)、Tシャツ罰は受け取り後 24 時間ルール
6. **強制発注エンジン（逆比例＋ランダム）** で「作らないほど来週は地獄」を実装 (FR-2)
7. **レシピ配信**: 材料受け取り時にリポジトリ同梱の静的レシピ（手書きアミーゴ口調、キット種別 1:1）を配信 (FR-2.3、v1.7 で安全性のため動的生成を廃止)
8. **タコス欲刺激エージェント（控えめ FR-6.1 + 中程度 FR-6.2 の 2 層）** で AI が刺激を生成 (FR-6)（v1.8 で過激レイヤー削除）
9. **AWS サーバーレス中心**（規約上 AWS 必須、無料枠縛りは外し、Bedrock 等も自由に採用）
10. **Python** 実装、**AWS 内で完結**（v1.8 で外部 ChatGPT GPT スコープ削除）
11. **Security 拡張は無効**、**PBT 拡張は Partial**（仮）
12. **AI-DLC プロセスの実践そのものが審査対象** → aidlc-docs を丁寧に整備、Units Generation 必須

**審査差別化要素（v2.0 更新）**:
- **「Taco Tuesday の日本化」という文化移植ミッション**の明確さ（ビジネス意図の深度）
- **「認知の占有」という "人をダメにする" の定義**（ハッカソンテーマへの正面回答）
- **「最高じゃないですか」という幸せな降伏トーン**（ブラックコメディと愛情の共存）
- サルサ通知の音 UX
- 「Tシャツ発注済み」UI のシュールさ
- **AI を予測器ではなく"刺激生成器"として使う**設計の独自性（FR-6）
- 「食べないほど来週は地獄」という強制発注ロジックのコンセプト純度（FR-2.1）
- AI-DLC ワークフロー全ステージ実施の**プロセス遵守**
