# AI-DLC ハッカソン 採点ルーブリック

AWS Summit Japan 2026 AI-DLC ハッカソン 参加規約（審査基準）に基づく 12 軸の詳細採点基準。

---

## バンド定義（全軸共通）

| バンド | 点数 | 意味 |
|---|---|---|
| **Excellent** | 90-100 | 1 位を狙える水準。独自性 + 完成度 + 説得力がすべて揃っている |
| **Strong** | 75-89 | 上位入賞圏。1〜2 個の強い武器を持っている |
| **Adequate** | 60-74 | 書類通過水準。平均的で可もなく不可もなし |
| **Weak** | 40-59 | 通過リスクあり。改善しないと他チームに負ける |
| **Poor** | 0-39 | 失格・大幅減点リスク。即時対処必須 |

---

## 書類審査（4 軸）

### 軸 DS-1: ビジネス意図（Intent）の明確さ

**評価の問い**: 「何を作りたいか・なぜそれか」が初見のレビュアーに 5 分で伝わるか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | テーマとの整合性が鮮明。ユーザー行動の変化まで描けている。ビジネス/社会的文脈が独自性を持つ |
| Strong | 75-89 | 意図は明確。テーマ「人をダメにするサービス」との接続が説明できる |
| Adequate | 60-74 | 意図は伝わるが「なぜこのアイデアか」の独自性が薄い |
| Weak | 40-59 | 意図が散漫。複数のコンセプトが混在していてどれがコアか不明 |
| Poor | 0-39 | 意図が読み取れない。テーマとの関係が不明確 |

**「これがあると高得点」チェック**:
- [ ] Concept Statement（1〜2 文のエレベーターピッチ相当）が明文化されている
- [ ] 「なぜタコスか」「なぜ依存させるか」の設計思想が言語化されている
- [ ] ユーザーの行動変化（Before/After）が描かれている
- [ ] テーマ「人をダメにする」の解釈が他チームと差別化されている
- [ ] バージョン改訂履歴があり「なぜそう変わったか」が追跡可能

**「これが欠けると即減点」チェック**:
- [ ] Concept Statement がなく、ドキュメントの随所に断片的に散らばっている → -10
- [ ] テーマ適合を「タコスが好き」程度にしか説明できない → -15
- [ ] ビジネス意図とシステム設計が乖離している → -10

---

### 軸 DS-2: Unit 分解の適切さ

**評価の問い**: システムが適切な粒度・独立性で分解されており、実装に向けて進める状態か？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | 各 Unit が単一責務、依存関係が明示、Walking Skeleton 戦略が存在、FR・ストーリーとのトレーサビリティが完全 |
| Strong | 75-89 | Unit 分解は妥当。依存関係が概ね整理されている |
| Adequate | 60-74 | Unit は定義されているが責務が重複または曖昧な箇所がある |
| Weak | 40-59 | Unit 分解が粗すぎる or 細かすぎて実装困難 |
| Poor | 0-39 | Unit 分解がない or システムが一塊で分解されていない |

**「これがあると高得点」チェック**:
- [ ] 全 Unit に責務・主要 FR・主要ストーリーが明記されている
- [ ] Unit 間依存関係が可視化されている（マトリクス・図）
- [ ] Walking Skeleton など実装戦略が明確
- [ ] コードの配置場所（ディレクトリ構造）が設計済み
- [ ] Unit 外に出す要素（外部サービス等）の判断根拠が記録されている

**「これが欠けると即減点」チェック**:
- [ ] Unit 間の直接呼び合い（密結合）が設計に含まれている → -10
- [ ] Unit 分解とストーリーのマッピングがなく実装順序が不明 → -8
- [ ] 依存関係が未整理で循環依存の可能性 → -12

---

### 軸 DS-3: 創造性とテーマ適合性

**評価の問い**: 審査員が「おもしろい」「これは新しい」と感じるか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | テーマ解釈が独自。メカニズムが実際に「人をダメにする」ことを技術的に実現している。ユーモアや驚きがある |
| Strong | 75-89 | 創造性あり。テーマ適合はわかりやすい。ただし他チームも思いつく可能性が高い |
| Adequate | 60-74 | アイデアは普通。テーマとの接続が説明的すぎる |
| Weak | 40-59 | テーマ「人をダメにする」の解釈が浅い。依存や悪影響の設計が甘い |
| Poor | 0-39 | テーマと実質的に無関係 |

**「これがあると高得点」チェック**:
- [ ] テーマ「人をダメにする」を複数の依存ループで実現している（単一メカニズムでない）
- [ ] AI の使い方が「予測/推薦」という通常解釈ではなく独自の用途
- [ ] ユーモア・シュールさが要件レベルで保証されている（言葉遊びでなく仕組みで）
- [ ] 「脱出口をなくす」「逃げると状況が悪化する」設計
- [ ] 視覚的・体験的インパクトがデザインレベルで意図されている

**「これが欠けると即減点」チェック**:
- [ ] 「タコスが好き → 作れるようにする」というポジティブサービスに留まっている → -15
- [ ] AI が単なる便利ツールとして使われており「ダメにする」メカニズムに関与していない → -10
- [ ] 創造性がドキュメントに文字として書かれているだけで仕組みが伴っていない → -8

---

### 軸 DS-4: ドキュメントの品質

**評価の問い**: 外部のレビュアーが全体を把握できるか？情報密度と構造は適切か？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | README が入口として完璧。全成果物に章立て・図・トレーサビリティあり。初見で 10 分以内に全体把握可能 |
| Strong | 75-89 | README 以外は充実。全体像はやや手間がかかるが把握可能 |
| Adequate | 60-74 | ドキュメントは存在するが統合されておらず、どこを見れば良いか不明確 |
| Weak | 40-59 | ドキュメントが断片的。重要な設計判断の根拠が書かれていない |
| Poor | 0-39 | ドキュメントが実質的に存在しない、または読めない状態 |

**「これがあると高得点」チェック**:
- [ ] `README.md` がプロジェクト概要・コンセプト・ドキュメント導線・デモリンクを含む
- [ ] 全ドキュメントにバージョン履歴と更新理由が記録されている
- [ ] Mermaid / ASCII art などの視覚化が適切な箇所で使用されている
- [ ] FR・ストーリー・コンポーネント・Unit 間のトレーサビリティが完備
- [ ] `audit.md` に意思決定の完全な追跡ログがある

**「これが欠けると即減点」チェック**:
- [ ] `README.md` が空または 1 行のみ（書類審査で公開 GitHub URL を提出する以上、入口が空では致命的） → **-20**
- [ ] `.gitignore` が空 or プレースホルダー → -3
- [ ] 成果物間の参照関係がなく孤立したドキュメント群 → -8
- [ ] 図が一切ない（テキスト only で複雑なアーキテクチャを説明） → -5

## 予選（4 軸）

### 軸 P-1: MVP デモの完成度

**評価の問い**: 「タコ中」の中核体験が動いて見えるか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | Walking Skeleton が全 Unit を薄く繋ぎ、End-to-End でコア体験（注文→配送→罰）が動作する |
| Strong | 75-89 | コア Unit（U3 OrderEngine / U6 Punishment / U7 PWA）が動作。デモとして成立 |
| Adequate | 60-74 | 一部 Unit のみ動作。主要な「ダメにするループ」が途切れている |
| Weak | 40-59 | ハードコードされた画面があるだけで「動作する」とは言えない |
| Poor | 0-39 | デモが存在しない、またはクラッシュする |

---

### 軸 P-2: AI-DLC プロセスの実践と工夫

**評価の問い**: INCEPTION → CONSTRUCTION を AI-DLC の手順通りに実践しており、工夫の跡が見えるか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | 全ステージが完走。意思決定の変遷（バージョン改訂・判断の理由）が audit.md に完全記録。独自の工夫（スキップ理由・応用）が説明できる |
| Strong | 75-89 | 主要ステージが完走。工夫の説明ができる |
| Adequate | 60-74 | ステージを踏んでいるが形式的で深みがない |
| Weak | 40-59 | 一部ステージを飛ばしており、成果物に抜けがある |
| Poor | 0-39 | AI-DLC プロセスがほぼ使われていない |

---

### 軸 P-3: 創造性とテーマ適合性（予選）

書類審査 DS-3 と同軸。実装された「ダメにするメカニズム」が動いているかを追加評価。

### 軸 P-4: プレゼンテーション

**評価の問い**: 15 分で審査員が本質を理解できるか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | ストーリーが明確（Problem → Solution → Demo → AI-DLC プロセス → 差別化）。デモが映える。時間通り |
| Strong | 75-89 | 内容は伝わるが構成や緩急に改善余地 |
| Adequate | 60-74 | 情報過多または不足。審査員の理解を前提にしすぎている |
| Weak | 40-59 | デモが動かないリスクがある。プレゼン構成が散漫 |
| Poor | 0-39 | 発表が成立しない |

## 決勝（4 軸）

### 軸 F-1: デモと完成度

**評価の問い**: AWS 上で本当に動いているか？ユーザー体験として完成しているか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | AWS にデプロイ済み。全 Unit が繋がり End-to-End で「ダメにするループ」が体験できる。エラーなし |
| Strong | 75-89 | AWS デプロイ済み。主要フロー動作。一部エッジケースは未対応 |
| Adequate | 60-74 | デプロイはされているが機能が不完全 |
| Weak | 40-59 | ローカル動作のみ、またはデモ動画のみ |
| Poor | 0-39 | AWS 上に動作するものが存在しない（規約要件未達・失格リスク） |

### 軸 F-3: 創造性とビジネス価値

**評価の問い**: 創造性が「ビジネスとして成立しうる」レベルまで昇華されているか？

| バンド | 点数 | 基準 |
|---|---|---|
| Excellent | 90-100 | ユニークな発想 + 実際に課金・収益化できそうな仕組み + 市場規模の説明 |
| Strong | 75-89 | 創造性あり。ビジネス文脈の説明は薄いが説得力はある |
| Adequate | 60-74 | 創造性 or ビジネス価値のどちらかが弱い |
| Weak | 40-59 | おもしろいが「ビジネス」として話せるものがない |
| Poor | 0-39 | どちらも弱い |
